# Google Workspace CLI Setup

## Goal

Enable the `gws` CLI locally and make it usable from Pylon agents through shell-enabled roles and a local skill.

## Current Local State

- Installed binary: `~/.cargo/bin/gws`
- Wrapper script for repo usage: `ui/scripts/gws`
- Installed version target: `0.13.2`

## Installation Path Used

The current machine does not have `npm`, `brew`, or Rust available, so the setup uses the official GitHub release tarball for Apple Silicon:

- Release: `https://github.com/googleworkspace/cli/releases/tag/v0.13.2`
- Asset: `gws-aarch64-apple-darwin.tar.gz`

## Verification Commands

```bash
~/.cargo/bin/gws --version
./ui/scripts/gws --version
./ui/scripts/gws auth status
```

## Authentication Options

### Option 1: Interactive local login

Use this when you have an OAuth desktop client JSON ready.

1. Create a Google Cloud project and OAuth Desktop client
2. Save the client JSON to:
   `~/.config/gws/client_secret.json`
3. Run:

```bash
./ui/scripts/gws auth login
```

### Option 2: `gws auth setup`

Use this only when `gcloud` is installed locally.

```bash
./ui/scripts/gws auth setup
./ui/scripts/gws auth login
```

### Option 3: Credentials file for headless agents

Use this for CI, service accounts, or a pre-exported credential bundle.

```bash
export GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE=/path/to/credentials.json
./ui/scripts/gws auth status
```

### Option 4: Delegated service account for Google Workspace organizations

Use this when a Google Workspace admin wants Pylon agents to act on behalf of a user through domain-wide delegation.

1. Create a service account in the target Google Cloud project
2. Enable `Domain-wide delegation` on that service account
3. Create and download a JSON key
4. Save the key to:

```text
~/.config/gws/service-account.json
```

5. In Google Workspace Admin Console, authorize the service account client ID for the exact scopes you intend to use
6. Export the delegation environment variables before running `./ui/scripts/gws`

Recommended initial Pylon scope set:

```text
https://www.googleapis.com/auth/drive.readonly,https://www.googleapis.com/auth/gmail.modify,https://www.googleapis.com/auth/calendar,https://www.googleapis.com/auth/documents,https://www.googleapis.com/auth/spreadsheets,https://www.googleapis.com/auth/chat.spaces.readonly,https://www.googleapis.com/auth/chat.messages,https://www.googleapis.com/auth/admin.directory.user.readonly
```

Why this set:

- `drive.readonly`: search and read Drive files without granting global Drive write access
- `gmail.modify`: read mail, label it, and manage mailbox workflows without permanent delete scope
- `calendar`: create and update events for scheduling workflows
- `documents`: create and edit Google Docs generated by agents
- `spreadsheets`: create and edit Sheets used for tracking and reporting
- `chat.spaces.readonly`: inspect target Chat spaces safely
- `chat.messages`: post and read messages in Chat spaces
- `admin.directory.user.readonly`: resolve users in the Workspace directory for routing and automation

Recommended expanded Pylon scope set for file management:

```text
https://www.googleapis.com/auth/drive,https://www.googleapis.com/auth/gmail.modify,https://www.googleapis.com/auth/calendar,https://www.googleapis.com/auth/documents,https://www.googleapis.com/auth/spreadsheets,https://www.googleapis.com/auth/chat.spaces.readonly,https://www.googleapis.com/auth/chat.messages,https://www.googleapis.com/auth/admin.directory.user.readonly
```

Use the expanded set when agents need to:

- create folders
- move or rename existing Drive files
- upload local files to Drive
- download Drive files to the local filesystem
- clean up temporary artifacts after automation runs

```bash
export GWS_DELEGATED_SERVICE_ACCOUNT_FILE="$HOME/.config/gws/service-account.json"
export GWS_DELEGATED_SUBJECT="user@your-domain.example"
export GWS_DELEGATED_SCOPES="https://www.googleapis.com/auth/drive.readonly,https://www.googleapis.com/auth/gmail.modify,https://www.googleapis.com/auth/calendar,https://www.googleapis.com/auth/documents,https://www.googleapis.com/auth/spreadsheets,https://www.googleapis.com/auth/chat.spaces.readonly,https://www.googleapis.com/auth/chat.messages,https://www.googleapis.com/auth/admin.directory.user.readonly"
./ui/scripts/gws drive files list --params '{"pageSize": 5}'
```

The wrapper script mints a short-lived delegated access token and passes it to `gws` through `GOOGLE_WORKSPACE_CLI_TOKEN`.

### Option 4B: Keyless delegated impersonation when JSON keys are blocked

If your organization prevents downloading service account JSON keys, use the IAM Credentials API path instead.

Requirements:

- The service account already has domain-wide delegation enabled
- The service account client ID is authorized in Google Workspace Admin Console for the scopes you need
- The operator identity has `roles/iam.serviceAccountTokenCreator` on the service account
- You have a valid operator OAuth access token for Google Cloud

Example:

```bash
export GWS_DELEGATED_SERVICE_ACCOUNT_EMAIL="pylon-244@ml-server-489002.iam.gserviceaccount.com"
export GWS_OPERATOR_ACCESS_TOKEN="ya29..."
export GWS_DELEGATED_SUBJECT="user@your-domain.example"
export GWS_DELEGATED_SCOPES="https://www.googleapis.com/auth/drive.readonly,https://www.googleapis.com/auth/gmail.modify,https://www.googleapis.com/auth/calendar,https://www.googleapis.com/auth/documents,https://www.googleapis.com/auth/spreadsheets,https://www.googleapis.com/auth/chat.spaces.readonly,https://www.googleapis.com/auth/chat.messages,https://www.googleapis.com/auth/admin.directory.user.readonly"
./ui/scripts/gws drive files list --params '{"pageSize": 5}'
```

This avoids storing a long-lived service account private key on disk.

### Option 5: Pre-obtained access token

```bash
export GOOGLE_WORKSPACE_CLI_TOKEN="ya29..."
./ui/scripts/gws drive files list --params '{"pageSize": 5}'
```

## Pylon Agent Usage

The repo-local skill `google-workspace-cli` tells agents to:

- use `./ui/scripts/gws`
- check auth state before acting
- prefer `--dry-run` for write operations
- keep output structured JSON
- use delegated env vars when acting through a Workspace-admin-approved service account

The seeded `workspace-automation-engineer` and shell-capable integration roles are wired to that skill in `ui/scripts/start_backend.py`.

## Notes

- The upstream repo explicitly states this is **not an officially supported Google product**.
- For containerized Linux runtimes, install the Linux release asset instead of the Darwin build used on this workstation.
- Domain-wide delegation still requires Google Workspace Admin Console approval for every OAuth scope you grant to the service account.
